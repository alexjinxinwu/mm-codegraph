## Why

工具的入口是一次搜索:用户手里只有一个外部标识(一个 commandId 或一个 flowId),
需要一个地方把它输进去、得到一个可探索的起点。在此之前没有任何界面,后端的
/resolve 与 /expand 也就无人触达。本能力提供这道门:一个持久的外壳布局,加上
把标识解析成根节点的搜索。

搜索的结果不是单一的。/resolve 会返回三态:查无此物、唯一命中、多个候选待消歧。
界面必须把这三态分别表达清楚——空状态提示、直接落根、候选选择器——而不是只处理
「成功」一种情况。把三态翻译成三种界面表现,是本能力的核心职责。

外壳只负责「把第一个节点放进画布」。节点一旦成为种子,后续的渲染与点击展开由
画布能力接手。这样搜索与画布解耦:搜索关心「从哪开始」,画布关心「怎么往下走」。
本能力依赖 entry-resolution-api(/resolve),不直接触碰 /expand。

## What Changes

- 新增浏览器外壳:全局布局,划分搜索栏、主画布区域、状态区(空 / 加载 / 错误)。
- 新增 schema 选择器,作为后续所有查询的作用域。
- 新增搜索控件:kind 选择(commandId / flowId,镜像 ENTRY_RESOLVERS 的已注册入口)
  + value 输入,提交后调用 /resolve。
- 三态界面表现:
  - found → 把根节点作为种子交给画布区域。
  - multiple → 渲染候选选择器,用户选定后该候选成为种子。
  - notFound → 渲染空状态提示,不报错。
- 加载态与错误态(网络 / 服务端错误,区别于 notFound)。

## Deferred(后续 change)

- 图的渲染与点击展开(canvas 能力,消费 /expand)。
- 节点详情面板。
- 搜索历史 / 最近访问。
- 不指定 kind 的跨入口统一搜索。
- URL 深链与可分享状态、键盘快捷键。

## Impact

- Affected specs: 新增 `browser-ui` capability。
- Affected code: 前端外壳布局组件、schema 选择器、搜索控件、三态渲染、画布种子
  接口(占位,实际渲染由后续 canvas 能力实现)。
- 依赖 entry-resolution-api(/resolve);本能力为只读消费,无数据库交互。
